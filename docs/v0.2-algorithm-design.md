# v0.2 高智能感知推荐算法设计文档

## 1. 设计背景

在 v0.1 版本的组合推荐中，算法统一使用 `MTokens/月` 作为唯一的需求与额度评估指标。这导致了“高低档订阅混淆”的问题：系统可能推荐大额度但模型能力平庸的组合去覆盖用户的极高开发需求，或将高难度的编程任务分配给低智能的免费模型额度。

v0.2 引入了**“高智能模型需求比例”**以及**“模型能力分档”**机制，以精确区分用户的任务复杂度和不同订阅的智能供给等级。

---

## 2. 核心改变

### 2.1 高智能需求比例输入

用户在输入端选择高智能需求占比（`highIntelligenceRatioPreset`），算法内部将其映射为数值占比（`highRatio`）：

| 档位 | 比例 | 适用场景 |
| :--- | :---: | :--- |
| **低 (low)** | 20% | 日常轻度对话、内容修正与简单文本操作 |
| **中 (medium)** | 50% | 普通开发、中等难度问答与调试混合场景 (默认) |
| **高 (high)** | 80% | 复杂系统架构、深度重构与高频开发场景 |
| **极高 (extreme)** | 95% | 全依赖 Reasoning 模型、大规模 Agent 项目修改 |

---

### 2.2 模型能力分档数据 (`model-tiers.json`)

系统为核心复杂能力维度（`agentCoding / backend / debugging / codeReview / research`）定义模型能力等级（`S / A / B / C / D`）。

高智能额度折算乘数（`highIntelligenceQuotaFactor`）：

$$Factor_{tier} = \begin{cases} 1.0 & \text{tier = S} \\ 0.85 & \text{tier = A} \\ 0.45 & \text{tier = B} \\ 0.15 & \text{tier = C} \\ 0.0 & \text{tier = D} \end{cases}$$

#### 档位判定逻辑：
- 如果一个计划在 `model-tiers.json` 中配置了具体核心维度的等级，则应用该配置；
- 若未配置且能力属于核心高复杂度维度（如开发/排错/架构），默认兜底等级为 **`C`** (折算乘数 0.15)；
- 其它低复杂度维度默认兜底为 **`B`** (折算乘数 0.45)。

---

### 2.3 双层额度分配算法

计算组合的能力评分时，需求分配过程升级为双层贪心分配法：

#### Layer 1：高智能需求优先分配
- 统计用户在各个核心维度上的高智能需求：
  $$HighDemand_c = TotalDemand \times Weight_c \times HighRatio$$
- 按权重降序处理核心维度，将 $HighDemand_c$ 优先分配给该维度下智能等级最高（S级优先，依次降序）的计划额度。
- 每步分配的可用高智能额度受计划的高智能额度上限与物理剩余额度（`textQuota`）的双重约束：
  $$Allocated = \min(HighDemand_c, HighQuota_{plan, c}, RemainingPhysical_{plan})$$
- 扣减物理剩余额度并计入高智能分配结果。

#### Layer 2：普通需求分配
- 统计用户在所有维度上的普通需求：
  $$GeneralDemand_c = TotalDemand \times Weight_c \times (1 - HighRatio) \quad \text{(核心维度)}$$
  $$GeneralDemand_c = TotalDemand \times Weight_c \quad \text{(非核心维度)}$$
- 按照得分高低，将普通需求分配到各计划的剩余物理额度上，直至少数低配额度耗尽。

---

### 2.4 双重覆盖过滤条件

在组合过滤阶段，候选组合必须**同时**满足两项硬性指标，否则不予默认推荐：

1. **总用量覆盖率** $\ge 90\%$：
   $$TotalTextQuota \ge TotalDemand \times 0.9$$
2. **高智能有效额度覆盖率** $\ge 90\%$：
   $$HighIntelligenceEffectiveQuota \ge HighDemand \times 0.9$$

---

## 3. 已知限制与后续展望

- **缓存模型简化**：继续采用 95% 的静态缓存命中率假设，后续版本可根据模型分档引入动态缓存系数。
- **混合分档处理**：对于像 Poe 和 OpenRouter 这种聚合 API 套餐，其分档等级按高智能路由兜底（A级）计算，未做子模型动态精细拆分。
