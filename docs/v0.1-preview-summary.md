# SubPlan v0.1 预览版整理

更新时间：2026-07-01

## 版本定位

v0.1 是一个轻量网页工具的预览版，目标是验证核心推荐链路是否成立：

- 用户输入预算、月度用量、主要用途、次要用途、地区和支付条件。
- 用户可以选择已有订阅。
- 系统基于订阅价格、估算额度、能力评分和组合关系，给出订阅组合建议。
- 结果页解释每个组合为什么被推荐，并提示预算、覆盖率、额度来源和注意事项。

这个版本不追求生产级数据完全准确，也不追求复杂算法最优解。重点是让推荐路径可跑通、结果可解释、交互可验证。

## 已实现功能

### 输入表单

- 月预算：按总月成本理解，已有订阅也计入预算。
- 预算宽松度：严格、默认 15%、宽松 25%。
- 月度用量：以 MTokens-equivalent 表示。
- 用量估算弹窗：按使用时长、对话频次和 Agent/长文件/调试倍率估算。
- 主要用途和次要用途：由 `data/presets.json` 映射为能力权重。
- 地区、是否接受 API 计费、是否有外币卡。
- 附加能力需求。
- 已有订阅选择。

### 推荐结果

- 分为最推荐、省钱、高性能和其它候选。
- 每张卡片展示组合名称、单项状态、价格、总额度、能力分、综合分、覆盖率和预算状态。
- 单项状态支持：
  - 已拥有
  - 建议新增
  - 建议升级自某套餐
- 已有订阅会作为基础组合一起参与计算。
- 同品牌高档套餐可以作为升级候选，但会替换原有低档，不与原有低档叠加。
- 免费主订阅不作为主动新增候选。
- 卡片底部说明里补充额度估算：
  - 组合总额度
  - 单项套餐估算额度
  - 来源为社区统计与 AI 估算，不承诺 100% 准确

### 数据管理

- `data/plans.json`：订阅套餐、价格、地区、支付和来源信息。
- `data/quotas.json`：套餐额度估算。
- `data/scores.json`：能力维度评分。
- `data/relations.json`：套餐之间的互补/重叠关系。
- `data/presets.json`：用途预设和能力标签。
- `data/api-options.json`：API 补位候选。
- `data/fx-rates.json`：汇率。
- `/admin` 提供本地数据编辑界面。
- `/api/admin` 在开发/localhost 环境允许写入数据文件。

## 当前推荐算法口径

### 预算

预算按组合总月成本计算。

如果用户选择已有订阅，它们会计入总成本。推荐新增或升级套餐时，本质上使用的是：

```text
可用空间 = 用户预算 - 已有订阅费用
```

升级场景按替换计算。例如已有 Kimi 99，推荐升级 Kimi 199：

```text
总月成本 = 其它已有订阅 + Kimi 199
变更成本 = Kimi 199 - Kimi 99
```

预算状态按总月成本判断，而不是只按变更成本判断。

### 用量

用量以 MTokens-equivalent 表示。

当前额度不是官方固定 token 上限，而是基于公开信息、社区统计和 AI 估算的工作量口径。结果卡片会明确提示不承诺 100% 准确。

组合需要至少覆盖 90% 用户输入用量；覆盖率达到 100% 视为覆盖充足。

### 能力评分

用户用途会被转换为能力权重，例如：

- backend
- frontend
- agentCoding
- debugging
- chineseWriting
- research
- multimodal

算法按各能力权重拆分用户总用量，再把每个能力维度的需求优先分配给该维度评分最高且仍有额度的套餐。

能力分是额度约束后的加权质量分，不是简单平均分。

### 组合生成

无已有订阅时，默认最多枚举 3 个套餐的组合。

有已有订阅时：

- 已有订阅固定作为基础组合。
- 免费主订阅不会作为新增项。
- 同品牌低档/同档不会作为新增项。
- 同品牌更高档可作为升级项，并替换原有同品牌低档。
- 额外新增项最多枚举 2 个，避免组合爆炸和推荐过复杂。

### API 补位

API 不是主要推荐方式。

只有用户接受 API 计费时，API 才作为预算剩余空间内的补位候选参与组合。当前 v0.1 只做轻量处理，不把 API 计费作为主要推荐路径。

## 已知限制

- 套餐额度估算仍有不确定性，尤其是动态限额、模型切换、缓存命中率和不同使用方式差异。
- 能力评分目前仍偏连续分数，部分模型之间的真实体验断层没有充分拉开。
- 高智能模型用量和普通冲量用量尚未拆分。
- API 补位逻辑较粗，只适合作为小预算剩余填补。
- `/admin` 是本地维护工具，不是生产级后台。
- 免费额度没有被当作正式推荐资源；这符合 v0.1 的简化原则，但也意味着用户真实可用的零散免费额度不会被精细计入。
- 当前不处理订阅支付层细节，例如能否稳定支付、是否需要境外账号、是否有区域限制变化。

## v0.2 高智能感知推荐算法实现与升级

v0.2 已经于 2026-07-01 正式实现。核心优化包括：

### 高智能模型需求比例 (已实现)
- 首页表单新增“高智能模型需求比例”分段选择（低：20%，中：50%，高：80%，极高：95%）。
- 推荐引擎内部将用户总用量（MTokens）拆分为高智能需求与普通/冲量需求。
- 强制要求组合必须**同时**满足**总覆盖率 $\ge 90\%$** 和 **高智能有效覆盖率 $\ge 90\%$** 双重阈值过滤。

### 模型能力分档 (已实现)
- 新增 `data/model-tiers.json` 数据集，定义主要套餐在 `agentCoding / backend / debugging / codeReview / research` 核心高复杂度维度上的等级（S/A/B/C/D）。
- 设定高智能有效折算折旧系数：S级折算 100%、A级 85%、B级 45%、C级 15%、D级 0%。

### 双层额度分配 (已实现)
- 重构额度贪心分配：第一阶段先用高智能有效额度饱和式覆盖高智能核心能力需求；第二阶段将普通需求以及未被高智能填满的剩余物理额度分配给其余用量维度。

### 管理后台更新 (已实现)
- 后台 `/admin` 新增“模型分档” JSON 代码在线编辑修改功能，打通动态读取与服务器实时持久化保存。

## 已知限制与后续展望
- 缓存命中率目前使用 95% 静态平均假设，未来可基于模型在不同高智能等级下的表现精细化配置。
- Poe/OpenRouter 等多模型聚合平台使用固定 A 级作为高智能平均权重，后续可支持按子模型配比拆分。
